# Plan 11-03 — Order Loading in Template Engine

## Objective
Staff can enter an order number in template-engine.html and have the book auto-assembled from the customer's uploaded photos. Template, page count, FP selections, and FP2 funny words all pre-populate from the order. Photos stream in and EXIF-sort runs automatically.

## Context
- @pages/template-engine.html — full engine; photo upload, EXIF sort, buildBookSequence, renderBook all exist
- `getOrderAssets` Cloud Function (Plan 11-02, must be complete first)
- GCS signed URLs expire after 1 hour — sufficient for a staff session

## Pre-requisite
Plan 11-02 deployed and tested. `getOrderAssets` returns correct response for a real order.

## Tasks

### Task 1 — "Load order" UI panel
Add a collapsible panel at the very top of the engine page, above the config bar. Collapsed by default (shows only "Load order ▾"). On expand:

```
Order number: [ AEV-___  ] [ Load ]
```

Style: same muted background as config bar. Input width ~120px. Button same as existing small buttons in the engine.

On Load button click: call `loadOrder(orderNumber)`.
Show inline status below the input: "Loading…" → "AEV-001 loaded — Kseniia M, 40 pages, 3 add-ons" on success → error message on failure.

Panel collapses after successful load (or staff can collapse manually).

### Task 2 — `loadOrder()` function
```js
async function loadOrder(orderNumber) {
  // 1. POST to getOrderAssets endpoint
  // 2. On error: show error, return
  // 3. Apply order config:
  //    - Set page count selector to order.pageCount (trigger existing chip selection logic)
  //    - Set template to order.templateName (Scribble for now; extend when more templates exist)
  //    - Check FP addon checkboxes matching order.fpSelections names
  //    - Rebuild book sequence: call buildBookSequence() + renderBook()
  // 4. Pre-populate FP2 text panel with order.fp2Words (if present)
  //    - Find the FP2 spread in the rendered book
  //    - Set its textPanel contenteditable text to fp2Words (each word on its own line)
  // 5. Load photos:
  //    - Filter files where fileType === 'main' → load into main photo pool
  //    - Filter files where fileType === 'cover' → load as cover photo
  //    - Filter files where fileType === 'special' → load into correct special pool by addonSlug
  //    - For each file: fetch(url) → blob → File object (preserve original filename)
  //    - Feed into existing processOneFile() pipeline (HEIC conversion, EXIF, orientation)
  //    - Process sequentially (same as manual upload — parallel HEIC conversion is broken)
  // 6. After all files processed: re-run EXIF sort → re-run assignPhotosToSpreads → re-render all spreads
  // 7. Update status line with result summary
}
```

**Progress feedback during load:**
Reuse the existing debug log panel for progress output:
- "Fetching order AEV-001…"
- "Loading 34 photos…"  
- "Processing photo 12/34…"
- "EXIF sort complete — 34 photos sorted"
- "Book assembled."

### Task 3 — Photo fetch + pipeline integration
Fetching photos from signed GCS URLs:
```js
const resp = await fetch(signedUrl);
const blob = await resp.blob();
const file = new File([blob], fileName, { type: blob.type });
```

Then pass `file` to the existing `processOneFile(file)` function unchanged. This handles HEIC conversion, EXIF extraction, orientation, low-res warnings, and pool assignment automatically.

For special photos: after processOneFile returns the photo object, assign it to `window.specialPhotos[addonSlug.toUpperCase()]` (e.g. `window.specialPhotos['FP1']`), then trigger the special photo render for that FP spread.

### Task 4 — FP selection wiring
The order returns `fpSelections` as an array of names (e.g. `['Birthday spread', 'Funny words']`). The engine config bar has FP checkboxes. Match by name:

```js
const FP_NAME_MAP = {
  'Birthday spread': 'FP1',
  'Funny words':     'FP2',
  'Favourite toy':   'FP3',
  'First steps':     'FP4',
  'Artwork gallery': 'FP5',
};
```

Check each matching checkbox, then call the existing FP toggle handler to update `window.selectedFPs` and trigger rebuild.

## Verify
- Enter a real order number → panel shows "Loading…" then success message
- Config bar shows correct page count and FP selections for that order
- Photo strip fills with customer photos, sorted by EXIF date
- Special FP photos appear in their dedicated upload zones
- FP2 text panel pre-populated with funny words (one per line)
- Book spreads render with photos auto-assigned
- Entering an unknown order number shows a clear error
- Manual photo upload still works after a loaded order (staff can supplement)
- No console errors during load
- checkpoint:human-verify — staff opens a real order, confirms photos and book match what customer submitted
